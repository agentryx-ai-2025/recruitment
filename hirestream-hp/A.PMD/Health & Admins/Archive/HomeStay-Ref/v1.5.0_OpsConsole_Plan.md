## Why This Sprint Exists

A 3-hour troubleshooting session on 7 Apr 2026 required 20+ manual terminal commands
across `pm2 logs`, `grep`, `psql`, and `df` to diagnose 3 stuck applicants. That same
session took 5 minutes of analysis once we knew where to look. This sprint makes
every one of those 20 commands a single click in the browser — visible to super_admin
without SSH, VPN, or terminal knowledge.

**The goal:** Anything you would run on the terminal to understand what the app is doing
must be visible on this dashboard. If it's not there yet, add it.

---

## Isolation Guarantee

This sprint is completely isolated:
- **No changes** to existing route files (except registering one new route in routes.ts)
- **No schema migrations** — all health data is derived at query-time (no new tables)
- **New files only:**
  - `server/routes/admin/health.ts` — all 12 API endpoints
  - `client/src/pages/admin/ops-console.tsx` — full dashboard page
- **New routes only:** `/api/admin/health/*` — protected by `super_admin` / `system_admin` / `admin` role check
- Route registered inside try/catch guard — if health.ts fails, existing routes are unaffected
- Access via `/admin/ops` (new URL)

## Dashboard Tabs

### TAB 1: Overview
- Process status (ONLINE/STOPPED/ERRORED)
- Uptime, restart count, PID
- Node.js version, app version
- Last deploy timestamp
- Node heap & RSS memory
- **External Dependencies panel:**
  - Database (SELECT 1 ping + latency)
  - DB Pool (total/idle/waiting connections)
  - HimKosh gateway (HEAD request + latency)
  - Session Store (session table check)
  - Upload Storage (directory access check)
  - SMTP configured? (presence only)
  - SMS configured? (presence only)
- **Active Sessions:** count by role, list of online officers
- **Smart Alerts banner:** all active alerts surfaced at top
- **Environment Variables:** presence check (values never shown)

### TAB 2: Resources
- **CPU:** current %, count, model, load avg (1m/5m/15m), sparkline
- **Memory:** system total/free/used %, Node heap/RSS/external, sparkline
- **Disk Storage:**
  - Root partition usage (df -k /)
  - Uploads directory size (du -sb)
  - Backups directory size (du -sb)
  - Amber warning at >75%, red at >90%
- **Node.js Runtime:**
  - Event loop delay (ms) — warn >50ms, critical >100ms
  - Active handles & active requests
  - PID, platform, architecture

### TAB 3: Pipeline (Application Health)
- **Application grid:** all services × status groups (Draft / Submitted / Under Scrutiny / With DTDO / Payment Pending / Approved / Rejected)
  - Homestay, Tourism Unit, Restaurant, Travel Agent, Guide, Photographer
- **Stuck Apps (auto-detected):**
  - Payment pending > 48h (with owner name, mobile, last update)
  - Submitted, no DA assigned > 5 days
  - Under scrutiny > 10 days
  - Old drafts > 7 days
- **Today's Activity:** status updates from midnight today
- **Payment Health:**
  - Count by status (pending/completed/failed)
  - Oldest pending age (hours)
  - Pending total (₹)
  - Monthly and yearly collected revenue
- **Certificate Expiry Watchlist:**
  - Expiring in 7 days (urgent list with names)
  - Expiring in 30 days (count)
  - Already expired count

### TAB 4: Backups
- Hooks directly into existing `/api/admin/backup/*`
- Scheduler status (running/stopped + schedule)
- Last backup: status, timestamp, error if failed
- Backup history table (last 15 backups)
- Link to full backup management console

## API Endpoints Implemented

All at `/api/admin/health/*`, all require `super_admin`/`admin`/`system_admin`.

```
GET  /api/admin/health/vitals
     → Process status, uptime, version, env check, node memory

GET  /api/admin/health/resources
     → CPU samples, RAM (system + node), disk (root + uploads + backups), Node.js runtime

GET  /api/admin/health/dependencies
     → DB ping, DB pool stats, HimKosh ping, session store, upload storage, SMTP/SMS presence

GET  /api/admin/health/sessions
     → Active session count, breakdown by role, list of online officers

GET  /api/admin/health/pipeline
     → App counts by service × status, stuck apps, today activity, payment health, cert expiry

GET  /api/admin/health/alerts
     → Pre-computed smart alert list (14 conditions)

GET  /api/admin/health/logs
     ?level=&module=&status=&appId=&limit=&offset=
     → Paginated Pino JSON log reader

POST /api/admin/health/lookup/search
     Body: { q: "name|mobile|appNumber" }
     → Search across homestay + tourism unit applications

GET  /api/admin/health/lookup/:id
     → Full application detail (app + payments)

GET  /api/admin/health/db/reports
     ?report=by_district|by_category|revenue|stuck_drafts|officer_activity|cert_expiry_30|payment_reconciliation
     → Pre-built read-only SQL reports

POST /api/admin/health/db/custom
     Body: { sql: "SELECT ..." }
     → Read-only SQL sandbox (super_admin/system_admin only)
       Safety: BEGIN + transaction_read_only + statement_timeout 5s + SELECT only

POST /api/admin/health/actions/restart
POST /api/admin/health/actions/clear-logs
     → PM2 restart / flush (super_admin only, requires confirmed: true)
```

## Smart Alerts (14 Conditions)

| Alert | Condition | Severity |
|-------|-----------|----------|
| DB unreachable | SELECT 1 fails | 🔴 Critical |
| Memory critical | Heap > 400 MB | 🔴 Critical |
| Memory high | Heap > 280 MB | 🟡 Warning |
| Disk critical | Root > 90% | 🔴 Critical |
| Disk warning | Root > 75% | 🟡 Warning |
| Stuck payments | Any payment_pending > 48h | 🟡 Warning |
| Backup failed | lastBackupStatus = failed | 🟡 Warning |
| No recent backup | lastBackupAt > 25h ago | 🟡 Warning |
| Certs expiring soon | Any cert expiry < 7 days | 🟠 Info |
| Crash-looping | >3 restarts in <1h | 🔴 Critical |

## Technical Implementation Notes

### CPU Sampling (Module-Level Singleton)
Runs on server startup, maintained in `server/routes/admin/health.ts` module scope.
Samples `os.cpus()` tick delta every 5s, keeps rolling 30-point history.
Memory sampled every 30s (heap + RSS + system free).
No new npm dependencies.

### PM2 Stats
Uses `child_process.exec('pm2 jlist')` — parses JSON, finds app by name regex `/hptourism/i`.
Avoids importing pm2 as a Node module (conflict risk).

### Custom SQL Sandbox
```typescript
BEGIN;
SET LOCAL transaction_read_only = on;
SET LOCAL statement_timeout = '5000ms';
// run user query
ROLLBACK; // always
```
Blocks: INSERT, UPDATE, DELETE, DROP, TRUNCATE, ALTER, GRANT, REVOKE, CREATE, REPLACE, COPY.
Max 500 rows. super_admin/system_admin only.

### Log Reading
Reads Pino JSON log file (from `appConfig.logging.file.path`), falls back to
`~/.pm2/logs/hptourism-out.log`. Parses lines as JSON, filters in-memory.
Up to 500 entries per request, reverse-chronological.
