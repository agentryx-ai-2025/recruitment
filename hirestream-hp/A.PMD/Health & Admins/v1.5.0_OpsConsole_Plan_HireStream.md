## Why This Module Exists

The HP Tourism project proved that a 3-hour terminal troubleshooting session can become 5 minutes of browser clicks. HireStream inherits this lesson from Day 1. For a government portal handling 5,000 concurrent users, overseas placements, and sensitive candidate PII — real-time visibility is not a luxury, it's a compliance requirement.

**The goal:** Anything you would run on the terminal to understand what the app is doing must be visible on this dashboard. If it's not there yet, add it.

**HireStream-specific justification:**
- Candidates track applications for overseas employment — stuck applications mean missed job opportunities abroad
- Recruitment agencies need timely verification — delays block legitimate hiring
- Admin (HPSEDC) needs oversight of the entire pipeline without SSH access to production servers
- ISO 27001 compliance requires audit trails and monitoring capabilities
- 99.9% uptime SLA demands proactive alerting, not reactive firefighting

---

## Isolation Guarantee

This module is completely isolated from core business logic:
- **No changes** to existing route files (except registering one new route in routes.ts)
- **No schema migrations** — all health data is derived at query-time (no new tables)
- **New files only:**
  - `server/routes/admin/health.ts` — all API endpoints
  - `client/src/pages/admin/ops-console.tsx` — full dashboard page
- **New routes only:** `/api/v1/admin/health/*` — protected by `admin` role check
- Route registered inside try/catch guard — if health.ts fails, existing routes are unaffected
- Access via `/admin/ops` (new URL)

## Dashboard Tabs

### TAB 1: Overview
- **Health Score** — 0–100 (Healthy / Degraded / Unhealthy / Critical), computed from active alerts
- **Smart Alerts Banner** — all active alerts surfaced at top, with snooze options (1h / 4h / 12h / 24h / 7d)
- Process status (ONLINE/STOPPED/ERRORED)
- Uptime, restart count, PID
- Node.js version, app version
- Last deploy timestamp
- Node heap & RSS memory
- **External Dependencies panel:**
  - Database (SELECT 1 ping + latency)
  - DB Pool (total/idle/waiting connections)
  - Session Store (session table check)
  - Upload Storage (`/data/uploads/` directory access check)
  - SMTP configured? (presence check only)
  - SMS Gateway configured? (presence check only)
  - UIDAI API reachable? (HEAD request + latency) — Phase 2
  - HIM Access SSO reachable? (HEAD request + latency) — Phase 2
  - DigiLocker API reachable? (HEAD request + latency) — Phase 2
- **Active Sessions:** count by role, list of online users (admin/agent/employer/candidate breakdown)
- **Environment Variables:** presence check (values never shown — security)

### TAB 2: Resources
- **CPU:** current %, count, model, load avg (1m/5m/15m), rolling sparkline
- **Memory:** system total/free/used %, Node heap/RSS/external, rolling sparkline
- **Disk Storage:**
  - Root partition usage (`df -k /`)
  - Uploads directory size (`du -sb /data/uploads`)
  - Backups directory size (`du -sb /data/backups`)
  - Logs directory size
  - Amber warning at >75%, red at >90%
- **Node.js Runtime:**
  - Event loop delay (ms) — warn >50ms, critical >100ms
  - Active handles & active requests
  - PID, platform, architecture

### TAB 3: Pipeline (Placement Health)
- **Application Pipeline Grid:** status groups across the full placement workflow
  - Columns: Submitted → Reviewed → Shortlisted → Interview Scheduled → Interview Done → Selected → Placed → Rejected
  - Rows: By agency (top 10 active agencies) + totals
- **Stuck Applications (auto-detected):**
  - Submitted, not reviewed > 5 days (with candidate name, email, agency)
  - Shortlisted, no interview scheduled > 7 days
  - Interview scheduled > 14 days ago, no result recorded
  - Selected but no placement recorded > 10 days
- **Today's Activity:** status updates from midnight today (new applications, status changes, new registrations)
- **Agency Verification Health:**
  - Pending verification count (agencies waiting > 3 days = warning)
  - Verified vs. unverified ratio
  - Recently revoked agencies
- **Recruitment Drive Health:**
  - Upcoming drives (next 7 days)
  - Drives pending admin approval
  - Past drives with no recorded outcomes
- **Grievance Health:**
  - Open grievances count
  - Oldest unresolved grievance age (days)
  - Grievances by category breakdown
- **Registration Trends:**
  - New candidates today / this week / this month
  - New agencies today / this week / this month
  - New job postings today / this week / this month

### TAB 4: Backups
- Scheduler status (running/stopped + cron schedule + next run)
- Last backup: status, timestamp, error if failed
- Backup history table (last 15 backups with size, status, age)
- Manual backup trigger button
- Download/delete backup files (admin only)

### TAB 5: Signals (Trends)
- Period filter: Today / 7 days / 30 days / Custom date range
- Application submission trends (line chart)
- Placement success rates over time
- Agency verification turnaround time
- Match score distribution (histogram)
- User registration trends by role
- API response time percentiles (p50/p95/p99)

### TAB 6: Logs
- Last 2MB of Winston JSON log file
- Filters: log level (info/warn/error/fatal), module, status code, user ID, free-text search
- View modes: Grouped / Timeline
- Expandable stack traces for error entries

### TAB 7: Log Query Panel (within Logs Tab)
- RegExp pattern search across logs (`-i` case-insensitive, `-v` invert match flags)
- 6 quick-example patterns:
  1. `"level":"error"` — all errors
  2. `auth.*fail` — authentication failures
  3. `POST /api/v1/jobs` — job posting activity
  4. `matchScore` — match algorithm activity
  5. `agency.*verif` — agency verification events
  6. `upload.*fail` — file upload failures
- Raw + parsed output; expandable JSON results

### TAB 8: Lookup
- Search by candidate name, email, phone, or application ID
- Search by agency name or license number
- Search by job title or employer name
- Detail side panel shows:
  - Full candidate profile + application history
  - Full agency profile + job postings + verification status
  - Full job details + applicant list

### TAB 9: DB Explorer
- **8 Pre-built reports:**
  1. **By District** — candidates registered by district/location
  2. **By Agency** — applications per agency, placement rates
  3. **By Skill** — top skills in demand vs. available
  4. **By Placement Status** — full pipeline funnel
  5. **Stuck Applications** — applications not progressing
  6. **Agency Performance** — verification time, placement count, rating
  7. **Job Market** — active jobs by country, sector, salary range
  8. **Match Score Analysis** — average scores, distribution, accuracy
- **SQL Sandbox** — SELECT-only, 5-second statement timeout, max 500 rows, always rolled back
  - `super_admin` only
  - Blocks: INSERT, UPDATE, DELETE, DROP, TRUNCATE, ALTER, GRANT, REVOKE, CREATE, REPLACE, COPY

### TAB 10: System
- TCP connection breakdown (ESTABLISHED / TIME_WAIT / LISTEN / etc.)
- File descriptor gauge — current open / system limit
- Zombie process detector
- Listening ports list
- Top processes by CPU/memory
- **Process Actions** (admin only):
  - Restart application (2-step confirm)
  - Clear logs / flush PM2 log files (2-step confirm)

## API Endpoints

All at `/api/v1/admin/health/*`, all require `admin` role.

```
GET  /api/v1/admin/health/vitals
     → Process status, uptime, version, env check, node memory

GET  /api/v1/admin/health/resources
     → CPU samples, RAM (system + node), disk (root + uploads + backups + logs), Node.js runtime

GET  /api/v1/admin/health/dependencies
     → DB ping, DB pool stats, session store, upload storage, SMTP/SMS presence
     → (Phase 2: UIDAI, HIM Access SSO, DigiLocker reachability)

GET  /api/v1/admin/health/sessions
     → Active session count, breakdown by role (candidate/agent/employer/admin), list of online users

GET  /api/v1/admin/health/pipeline
     → Application counts by status, stuck apps, today's activity, agency verification health,
       recruitment drive health, grievance health, registration trends

GET  /api/v1/admin/health/alerts
     → Pre-computed smart alert list (16 conditions)

GET  /api/v1/admin/health/signals
     ?period=today|7d|30d&start=YYYY-MM-DD&end=YYYY-MM-DD
     → Trend data: submissions, placements, verifications, match scores, registrations, API p95

GET  /api/v1/admin/health/logs
     ?level=&module=&status=&userId=&search=&limit=&offset=
     → Paginated Winston JSON log reader

POST /api/v1/admin/health/logs/grep
     Body: { pattern: "regex", flags: "-i", limit: 200 }
     → In-process RegExp search across log file

POST /api/v1/admin/health/lookup/search
     Body: { q: "name|email|phone|appId|agencyName|licenseNumber" }
     → Search across candidates, agencies, employers, jobs, applications

GET  /api/v1/admin/health/lookup/:type/:id
     → Full detail: candidate (profile + apps + documents), agency (profile + jobs + placements),
       job (details + applicants), application (full chain)

GET  /api/v1/admin/health/db/reports
     ?report=by_district|by_agency|by_skill|by_placement_status|stuck_apps|agency_performance|job_market|match_scores
     → Pre-built read-only SQL reports

POST /api/v1/admin/health/db/custom
     Body: { sql: "SELECT ..." }
     → Read-only SQL sandbox (admin only)
       Safety: BEGIN + transaction_read_only + statement_timeout 5s + SELECT only

POST /api/v1/admin/health/actions/restart
     Body: { confirmed: true }
     → PM2 restart (admin only, requires confirmed: true)

POST /api/v1/admin/health/actions/clear-logs
     Body: { confirmed: true }
     → PM2 flush logs (admin only, requires confirmed: true)
```

## Smart Alerts (16 Conditions)

| # | Alert | Condition | Severity |
|---|-------|-----------|----------|
| 1 | DB unreachable | SELECT 1 fails | Critical |
| 2 | Memory critical | Node heap > 400 MB | Critical |
| 3 | Memory elevated | Node heap > 280 MB | Warning |
| 4 | Memory leak trend | Heap increasing monotonically over last 6 samples (3 min) | Warning |
| 5 | Event loop blocked | Event loop delay > 100 ms | Critical |
| 6 | Event loop slow | Event loop delay > 50 ms | Warning |
| 7 | Disk critical | Root partition > 90% | Critical |
| 8 | Disk low | Root partition > 75% | Warning |
| 9 | Stuck applications | Any application submitted > 5 days with no review | Warning |
| 10 | Agency verification backlog | Any agency pending verification > 3 days | Warning |
| 11 | No recent backup | No backup in > 25h | Warning |
| 12 | Backup failed | Last backup status = failed | Warning |
| 13 | Grievance backlog | Any open grievance > 7 days unresolved | Info |
| 14 | Process crash-looping | > 3 restarts in < 1h | Critical |
| 15 | File descriptors critical | FD usage > 90% of system limit | Critical |
| 16 | Upload storage full | Uploads directory > 80% of allocated space | Warning |

Each alert includes:
- Recommended action text
- Direct link to the relevant tab for investigation
- Snooze options: 1h / 4h / 12h / 24h / 7d (scaled by severity)

## Technical Implementation Notes

### CPU Sampling (Module-Level Singleton)
Runs on server startup, maintained in `server/routes/admin/health.ts` module scope.
Samples `os.cpus()` tick delta every 5s, keeps rolling 30-point history.
Memory sampled every 30s (heap + RSS + system free).
No new npm dependencies.

### PM2 Stats
Uses `child_process.exec('pm2 jlist')` — parses JSON, finds app by name regex `/hirestream/i`.
Avoids importing pm2 as a Node module (conflict risk).

### Custom SQL Sandbox
```typescript
BEGIN;
SET LOCAL transaction_read_only = on;
SET LOCAL statement_timeout = '5000ms';
// run user query
ROLLBACK; // always
```
Blocks: INSERT, UPDATE, DELETE, DROP, TRUNCATE, ALTER, GRANT, REVOKE, CREATE, REPLACE, COPY.
Max 500 rows. Admin only.

### Log Reading
Reads Winston JSON log file (from logger config path), falls back to
`~/.pm2/logs/hirestream-out.log`. Parses lines as JSON, filters in-memory.
Up to 500 entries per request, reverse-chronological.
