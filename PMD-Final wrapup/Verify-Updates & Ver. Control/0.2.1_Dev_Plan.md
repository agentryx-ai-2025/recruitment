## What ships in v0.2.1

### 1. Admin Roadmap Dashboard (PRIMARY)

**Where:** new admin-only route `/admin/roadmap` (also linked from header dropdown for admin / delivery roles).

**What it shows:**
- Header strip: product name, current version, vision sentence, overall progress (X% shipped, Y in flight, Z planned across N total features)
- Phase 0 (baseline) summary card — 38 shipped features, click to expand
- Phase 1-9 cards in sequence:
  - Phase number, theme, effort estimate
  - Status badge (shipped / in progress / planned / future)
  - Visible-ship description (the headline value)
  - Per-phase progress bar
  - Expandable feature list with status icons (✅ 🟡 ⚪ 🔵)
- Aggregate roll-up table at the bottom (matches the table in `Roadmap_Dev_Checklist.md`)

**Why it matters:**
- Customer-facing scoreboard — point a potential customer at a single URL and they see what's built, what's coming, what could be added for them
- Dev-team alignment — engineers always know what's on deck without re-reading the roadmap MDs
- Internal momentum — visualises the % done and motivates the team

**Anchor test:** ✓ Pure additive — new admin-only route, doesn't touch existing matrix UX
**Hub test:** ✓ Surfaces the strategic plan that drives every phase
**No-training test:** ✓ Single-page visualisation, self-explanatory

### 2. Activity Feed (SECONDARY — Phase 1 starter)

**Where:** new collapsible panel on the project view, just above the Issues log section.

**What it shows:**
- Last 20 events from `audit_log` for that project
- Each row: timestamp, who, action, target (with link to row if applicable)
- Filter chips: "All" / "Sign-offs" / "Sprints" / "Feedback" / "Issues"
- Expand/collapse — collapsed by default for non-admin (don't add visual clutter), expanded by default for admin

**Why it matters:**
- HPSEDC team starts this afternoon. After their session, admin can see what they did at a glance.
- Already-collected data (`audit_log` is populated on every login + signoff change). Just rendering.
- Phase 1 deliverable arriving early without the rest of Phase 1's structural work.

**Anchor test:** ✓ Sits inside existing project view as a sibling section
**Hub test:** ✓ Captures cross-event view of project activity
**No-training test:** ✓ Reads chronologically, plain English

## Implementation plan

### Roadmap Dashboard

**Files to add:**

1. `agentryx-verify/shared/roadmap.ts` — typed roadmap data structure. Single source of truth that mirrors `Roadmap_Dev_Checklist.md`. Each phase has features with status enum. Hand-synced with the markdown for now; future work parses the markdown directly.

2. `agentryx-verify/client/src/pages/Roadmap.tsx` — admin-only dashboard page. Reads from `shared/roadmap.ts`, renders phase cards + roll-up table.

**Files to modify:**

3. `agentryx-verify/client/src/App.tsx` — register `/admin/roadmap` route.

4. `agentryx-verify/client/src/components/Header.tsx` — add "Roadmap" link visible to admin / delivery only.

**Access control:**
- Route loads, but if reviewer isn't admin / delivery, redirect to `/`
- Header link only renders for admin / delivery
- No backend changes needed (data is static, lives in client bundle)

**Visual style:**
- Reuse existing colours and Tailwind patterns (slate / agentryx-600 / amber / emerald / red / indigo)
- Phase cards in vertical stack with subtle accent border per status
- Donut chart for overall completion (reuse existing `Donut` component pattern from ProjectView)
- Match the existing amber "How to read" banner pattern for first-time visitor explanation

### Activity Feed

**Files to add:**

1. New component `agentryx-verify/client/src/components/ActivityFeed.tsx`

**Files to modify:**

2. `agentryx-verify/server/routes/projects.ts` — new endpoint `GET /api/projects/:slug/activity?limit=20&type=*` returns recent `audit_log` rows joined with reviewer name.

3. `agentryx-verify/client/src/lib/api.ts` — new `getActivity(slug, params)` method.

4. `agentryx-verify/client/src/pages/ProjectView.tsx` — render `<ActivityFeed slug={slug} me={me} />` before `<IssuesPanel />`.

**Backend logic:**
- Query `audit_log` where `project_id = project.id` (handle null project_id by joining via reviewer's signoff, or just filter — most actions write project_id)
- Join `reviewers` for name + organization
- Order by `created_at DESC LIMIT 20` (configurable via `?limit=`)
- Action grouping: prefix-based (`signoff.*`, `sprint.*`, `feedback.*`, `issue.*`, `auth.*`)
